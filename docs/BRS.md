# Business Requirements Specification (BRS)
## Consulting OS — منصة إدارة مكاتب الاستشارات

**Version:** 1.0  
**Date:** 2026-03-13  
**Status:** Draft

---

## 1. مقدمة

### 1.1 الغرض من الوثيقة
تحدد هذه الوثيقة المتطلبات التجارية الكاملة لمنصة **Consulting OS**، وهي منصة SaaS متعددة المستأجرين (Multi-Tenant) مصممة لإدارة مكاتب الاستشارات والبرمجة.

### 1.2 نطاق المشروع
المنصة تخدم ثلاثة مستويات رئيسية:
- **Super Admin:** يدير المنصة بالكامل وكل المكاتب المشتركة
- **المكتب:** يدير مشاريعه وفريقه وعملاؤه
- **العميل:** يتابع مشاريعه ويتواصل مع المكتب

### 1.3 الأطراف المعنية (Stakeholders)

| الطرف | الدور |
|---|---|
| Super Admin | مالك المنصة، يدير الاشتراكات والمكاتب |
| صاحب المكتب (Owner) | يدير المكتب بالكامل |
| المدير (Manager) | يدير المشاريع والفريق |
| الموظف (Employee) | ينفذ المهام المسندة إليه |
| صاحب الشركة العميلة (Client Owner) | يتابع مشاريع شركته ويدير موظفيه |
| موظف العميل (Client Employee) | يتابع المشروع حسب الصلاحيات الممنوحة له |

---

## 2. أهداف العمل

### 2.1 الهدف الرئيسي
توفير نظام مركزي يربط المكتب بعملائه وفريقه في مكان واحد، يحل محل الاعتماد على WhatsApp وExcel والأوراق.

### 2.2 الأهداف الفرعية

- **BR-01:** تمكين المكاتب من إدارة مشاريعها ومهامها وفريقها من منصة واحدة
- **BR-02:** منح العملاء بوابة شفافة لمتابعة مشاريعهم في الوقت الفعلي
- **BR-03:** أتمتة عملية إصدار الفواتير والدفع الإلكتروني
- **BR-04:** تمكين Super Admin من إدارة المكاتب والاشتراكات مركزياً
- **BR-05:** ضمان عزل كامل للبيانات بين المكاتب المختلفة

## 3. المتطلبات التجارية التفصيلية

### 3.1 إدارة المكاتب (Super Admin)

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-SA-01 | قدرة Super Admin على إنشاء مكتب جديد وتفعيله | Must Have |
| BR-SA-02 | قدرة Super Admin على إيقاف أو حذف مكتب | Must Have |
| BR-SA-03 | إدارة باقات الاشتراك (عدد المشاريع، الموظفين، التخزين) | Must Have |
| BR-SA-04 | عرض تقارير الأداء لكل المكاتب (إيرادات، نشاط، استخدام) | Must Have |
| BR-SA-05 | عدم قدرة Super Admin على الاطلاع على بيانات المشاريع أو العملاء داخل أي مكتب | Must Have |

### 3.2 إدارة المشاريع (المكتب)

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-PM-01 | إنشاء مشروع وربطه بعميل محدد | Must Have |
| BR-PM-02 | تقسيم المشروع إلى مراحل (Milestones) | Must Have |
| BR-PM-03 | إنشاء مهام داخل كل مرحلة وتعيينها لموظف | Must Have |
| BR-PM-04 | تتبع تقدم المشروع بنسبة مئوية تُحسب تلقائياً من المهام | Must Have |
| BR-PM-05 | عرض المهام على شكل Kanban Board | Must Have |
| BR-PM-06 | عرض المراحل على شكل Gantt Chart | Should Have |
| BR-PM-07 | تسجيل ساعات العمل على كل مهمة (Time Tracking) | Should Have |

### 3.3 إدارة الفريق (المكتب)

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-TM-01 | إضافة موظفين وتحديد أدوارهم (Manager / Employee) | Must Have |
| BR-TM-02 | تحديد صلاحيات كل دور على مستوى المشروع والمهام والبيانات المالية | Must Have |
| BR-TM-03 | عرض نسبة استغلال كل موظف (Utilization Rate) | Should Have |
| BR-TM-04 | عرض سجل ساعات العمل لكل موظف | Should Have |

### 3.4 بوابة العميل

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-CP-01 | يرى العميل فقط مشاريع شركته، لا مشاريع عملاء آخرين | Must Have |
| BR-CP-02 | قدرة Client Owner على إضافة موظفين من شركته | Must Have |
| BR-CP-03 | قدرة Client Owner على تحديد صلاحيات موظفيه (عرض فقط / تعليق ورفع ملفات) | Must Have |
| BR-CP-04 | قدرة العميل على إضافة تعليقات على المشاريع والمهام | Must Have |
| BR-CP-05 | قدرة العميل على رفع ملفات ومستندات مرتبطة بالمشروع | Must Have |
| BR-CP-06 | قدرة العميل على عرض الفواتير والدفع أونلاين | Must Have |

### 3.5 إدارة الفواتير

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-INV-01 | إنشاء فاتورة مرتبطة بعميل ومشروع | Must Have |
| BR-INV-02 | إضافة بنود متعددة مع حساب ضريبة وخصم | Must Have |
| BR-INV-03 | معاينة الفاتورة كـ PDF قبل الإرسال | Must Have |
| BR-INV-04 | إرسال الفاتورة للعميل مع إشعار تلقائي | Must Have |
| BR-INV-05 | تتبع حالة الفاتورة (Pending / Paid / Overdue) | Must Have |
| BR-INV-06 | تكامل مع بوابة دفع إلكتروني (Paymob) | Must Have |

### 3.6 التواصل والإشعارات

| الكود | المتطلب | الأولوية |
|---|---|---|
| BR-COM-01 | تعليقات داخل كل مشروع ومهمة | Must Have |
| BR-COM-02 | إشعارات فورية (Real-time) عند تغيير حالة المهمة أو المشروع | Must Have |
| BR-COM-03 | إشعارات بالإيميل عند إرسال فاتورة جديدة أو اقتراب موعد الاستحقاق | Must Have |
| BR-COM-04 | سجل نشاط (Activity Log) لكل مشروع | Should Have |

## 4. متطلبات غير وظيفية

### 4.1 الأمان
- عزل كامل للبيانات بين المكاتب (Multi-Tenant Isolation)
- المصادقة الثنائية (MFA) لجميع الأدوار
- تشفير البيانات الحساسة في قاعدة البيانات
- RBAC (Role-Based Access Control) على مستوى كل resource

### 4.2 الأداء
- وقت استجابة API لا يتجاوز 500ms في 95% من الطلبات
- يدعم النظام 1000 مستخدم متزامن كحد أدنى

### 4.3 التوافر
- Uptime لا يقل عن 99.5%
- نسخ احتياطي يومي تلقائي

### 4.4 قابلية الاستخدام
- واجهة عربية وإنجليزية مع دعم RTL
- متوافق مع الهواتف المحمولة (Responsive)

## 6. معايير القبول العامة

- كل مكتب يرى فقط بياناته ولا يصل لبيانات مكتب آخر
- كل دور يملك فقط الصلاحيات المحددة له في جدول الصلاحيات
- جميع العمليات المالية تُسجَّل في Audit Log لا يمكن تعديله
- أي إجراء حذف نهائي يتطلب تأكيداً صريحاً من المستخدم
